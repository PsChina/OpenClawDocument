# 第 1 章：认识 OpenClaw（深度解析）

> **分类**：数据分析 

## 1.1 OpenClaw 的诞生背景

OpenClaw 起源于开发者对现有 AI 助手的思考：大多数 AI 只能回答问题，却不能真正帮你干活。于是他们设计了一个 "有爪子的 AI"—— 既能理解你的意图，又能像你的手一样操作电脑。

## 1.2 核心架构详解

OpenClaw 由三个核心组件构成：

```
 聊天客户端
（飞书/微信/Telegram/Slack/Web控制台/命令行）
          |
          | WebSocket/HTTP
          v
  Gateway 服务
（端口 18789，负责路由消息、调用 skills、管理记忆）
          |
          |--------------------------|
          v                          v
  AI 模型层                       Skills 层
（支持多模型）                   （文件/浏览器/API等）
          |                          |
          |--------------------------|
          v
  记忆系统
（SQLite 存储身份/活跃/存档三层记忆）
```

- **Gateway**：所有消息的中转站，它接收聊天消息，调用 AI 模型理解意图，再调用对应的 Skills 执行操作，最后将结果返回给聊天客户端。Gateway 本身是一个 Node.js 服务，运行在本地 18789 端口。

- **Skills**：每个 Skill 是一个独立的模块，可以是用 JavaScript 编写的函数，也可以是调用外部 API 的脚本。它们被 Gateway 动态加载，AI 根据用户指令选择调用哪个 Skill。

- **记忆系统**：基于 SQLite 的向量数据库，存储用户与 AI 的交互历史，支持语义搜索。三层结构确保常用记忆始终在上下文，大量历史按需加载，极大节省 Token。

## 1.3 数据流向示例

当你对 AI 说："帮我整理下载文件夹"，数据流向如下：

1. 消息通过飞书 → Gateway
2. Gateway 将消息发送给 AI 模型（如混元、DeepSeek）
3. AI 理解意图，识别出需要调用 file-organizer Skill
4. Gateway 调用 file-organizer Skill，传入参数（下载文件夹路径）
5. Skill 执行文件整理操作，返回结果
6. Gateway 将结果通过飞书回复给你

整个过程在本地完成，只有 AI 模型调用可能涉及云端（如果你使用的是云端模型 API），但文件内容不会离开你的电脑。

## 1.4 Gateway 的工作原理

Gateway 是 OpenClaw 的大脑，它是一个基于 Express + WebSocket 的 Node.js 服务。主要组件包括：

- **消息路由器**：根据消息来源（飞书、微信等）解析发送者身份，调用对应的处理器。
- **AI 模型管理器**：管理多个模型提供商的连接池，支持流式输出和重试机制。
- **Skill 加载器**：启动时扫描 ~/.openclaw/skills/ 目录，动态加载所有 Skill 模块，并缓存以提高性能。
- **记忆客户端**：通过 HTTP 或 Unix Socket 与本地的记忆服务（Python 进程）通信，进行向量检索。
- **工作流引擎**：解析 YAML 工作流定义，调度步骤执行，处理条件分支和并行。

## 1.5 Skills 的注册与发现机制

每个 Skill 在安装时会在 ~/.openclaw/skills/ 下创建一个以 Skill 命名的目录，包含 index.js（主入口）和 package.json（依赖声明）。Gateway 启动时：

- 遍历所有 Skill 目录。
- 对每个 Skill，require() 其 index.js，获取 name、description、params 等元数据。
- 将 Skill 的执行函数包装在安全沙箱中（限制文件系统访问范围）。
- 将 Skill 元数据注册到内存中的技能列表，供 AI 调用时匹配。

AI 模型在决定调用 Skill 时，会看到所有技能的 name 和 description，以及参数说明。模型会根据用户指令选择合适的技能并填充参数。

## 1.6 记忆系统的实现细节（补充）

记忆系统是一个独立的 Python 进程，通过 agent-memory 命令行工具与 Gateway 交互。它使用：

- **SQLite 存储结构化数据**（事实、决策、身份信息）。
- **向量索引**：将文本转为向量（使用 all-MiniLM-L6-v2 等轻量模型），支持相似度搜索。
- **三层缓存**：身份层和活跃层常驻内存（由 Gateway 维护），存档层存储在 SQLite 中。

当你问 “你还记得我上次说的项目 deadline 吗？” 时，Gateway 将问题发给记忆服务，服务先搜索身份层和活跃层，若无结果则向量搜索存档层，返回最相关的结果。## 1.7 安全模型详解

OpenClaw 非常重视安全性，采取了多层防护：

- **文件系统沙箱**：Skill 只能访问配置中 security.allowedPaths 指定的目录（默认为用户主目录下的常见文件夹）。即使 Skill 恶意代码试图访问 /etc/passwd 也会被拒绝。
- **Skill 权限声明**：未来版本中，Skill 需要声明所需权限（如网络访问、文件写），安装时需用户确认。
- **消息来源验证**：对飞书、Telegram 等平台，验证 Webhook 签名，防止伪造消息。
- **敏感信息脱敏**：AI 模型调用时，自动检测并替换 API Key、密码等敏感信息（用占位符替换）。

## 1.8 性能指标

- **启动时间**：约 2-3 秒（加载 20 个 Skill）。
- **消息响应延迟**：AI 模型调用前约 50ms（路由 + 技能调度），AI 模型延迟取决于提供商。
- **记忆检索**：向量搜索 <100ms（10 万条记录）。
- **并发处理**：默认支持 5 个并发请求，可配置。

## 1.9 典型应用场景补充

- **个人知识问答**：将你的笔记、文档导入记忆系统，用自然语言提问，AI 从知识库中检索答案。
- **自动化测试**：用 AI 生成测试用例，并调用浏览器 Skill 执行 UI 测试。
- **智能家居控制**：通过集成 Home Assistant API，用语音在飞书控制灯光、空调。
- **代码审查助手**：每次 PR 自动拉取代码，AI 审查并留下评论。
