# 腾讯会议应用架构详解

本文档全面介绍腾讯会议（Tencent Meeting）跨平台客户端的技术架构，从底层技术栈、核心引擎到前端设计与业务支撑思路，重点突出 Electron 与 Web 技术在桌面端的运用。

## 一、底层技术栈

### 1. 平台与语言

- **桌面端（Windows、macOS）**
  - 主体采用 **Electron**：基于 Chromium + Node.js 的混合框架。
  - 前端使用 HTML5/CSS3/JavaScript，后端逻辑部分使用 TypeScript。
  - 对性能敏感的模块如音视频处理、屏幕共享使用 C++/Rust 编写的本地插件，通过 Node-API（N-API）或 Native Modules 调用。
- **移动端**
  - Android: Kotlin/Java + C/C++（通过 JNI 调用核心库）。
  - iOS: Objective-C/Swift + Objective-C++。
  - 侧重与桌面端共享核心引擎，UI 层走原生路线。

### 2. 核心音视频引擎

- **WebRTC** 为基础，腾讯在其上进行深度定制（自研 SDK 亦称 TRTC）。
- 编解码支持 H.264/H.265/VP8/VP9，音频支持 AAC/Opus。
- 包含强大的网络适配、FEC、NACK、码率与分辨率自适应机制。
- 屏幕分享和文件传输通过专用协议优化带宽与延迟。

### 3. 网络和服务器交互

- 使用 **TLS** 加密所有信令与媒体通道。
- 信令采用 WebSocket/HTTPS 进行双向实时通信。
- 通过负载均衡的云服务实现全球分布的会议转发与混流。

## 二、核心模块与中间层

### 1. Electron 桌面框架与安全隔离

- UI 页面在 Chromium 渲染进程中运行，逻辑和系统调用在 Node.js 主进程管理。
- **安全隔离机制**：采用 Context Isolation（上下文隔离）禁止渲染进程直接访问 Node.js 主进程；每个渲染器运行在沙箱（Sandbox）中，恶意脚本或漏洞利用被隔离。
- 利用 IPC（inter-process communication）实现渲染进程与主进程之间的数据传输，**仅通过预加载脚本（preload.js）指定白名单 API 窗口**。
- 主进程负责原生功能：系统通知、文件访问、permissionHandler 权限管理（如摄像头、麦克风）。

### 2. 原生模块与插件

- **Native Media Module**：基于 WebRTC C++ 源码编译的本地拓展，用于采集、渲染、编码和解码。
- **Screen Capture Module**：使用 **desktopCapturer API**（利用 Chromium 内置捕获机制），比 `getDisplayMedia()` 有更好的性能和窗口/应用三选择。macOS 需要 Accessibility 权限（系统级别 10+）；在 Windows 采用 DXGI 捕获桌面。
- **Audio Driver Module**：处理 3A 算法（AEC/NS/AGC）位置在音频输入阶段，随后接入 RtcEngine。

### 3. 业务逻辑层

- 会议房间管理、用户身份认证、权限控制（麦克风/摄像头、录制权限）。
- 弹性扩容的音视频混流服务，支持多人视频、屏幕共享、白板协作。
- 实时聊天、文件传输、会议纪要同步等功能模块。
- **录制与存储**：支持客户端本地录制（需要端口权限）和服务器 SFU 录制（需要管理员权限）。本地录制采用临时文件格式（可选 HEVC/H.265 或 H.264，不同算法重新编码）。服务器混流所有视频流合一、新存储粗粒仓库（输出中部分窗口）。

### 4. 数据与缓存

- 本地使用 LevelDB/SQLite 存储用户配置、会议记录和缓存文件。
- 配置采用 JSON 格式并支持远程下发、动态更新。

## 三、前端架构与交互

### 1. 框架与组件

- 桌面端 UI 使用 React + Redux/MobX 或企业内部组件库，实现组件化、可复用界面。
- 页面路由由 React Router 管理，会议页面与设置页面分离运行。
- 移动端则使用原生组件与 MVC/MVVM 架构。

### 2. Web/Hybrid 内容

- 部分功能（如快速帮助、插件市场、文档）通过内嵌 WebView 加载 Web 内容，与 Electron 具有同源优势。
- Electron 版本中使用预加载脚本（preload.js）暴露安全的 API 给渲染进程，以便调用本地功能。

### 3. 前端技术栈支持

- 使用 TypeScript 提高代码可维护性。
- UI 样式使用 Less/Sass 预处理器，配合 Webpack/Parcel 进行打包和按需加载。
- WebRTC API 在渲染进程中需要使用 **desktopCapturer** 而非标准 `getDisplayMedia()` 实现屏幕共享，结合本地插件增强兼容性。

### 4. 性能与优化

- 桌面端对内存和渲染进行监控，动态卸载非活跃模块以降低资源占用。
- 使用 lazy-loading 加载大型组件，如白板和录制回放。
- 通过 GPU 加速 Canvas 绘制，提高白板与远程控制体验。

## 四、业务支撑与应用逻辑

### 1. 会议场景与混流策略

- **小型会议**（5-20人）：采用 P2P 或 SFU 模式降低延迟。
- **大型会议/直播**（上千人）：使用云端混流/转码，将多个视频流合并发送到 CDN。当推流目标从单一主播切换到混流服务时，客户端平滑过渡；混流配置（分辨率、码率、布局）由服务器动态下发和调整。混流失败时自动降级为直播转发。
- **交互功能**：举手、投票、分组讨论、举牌等逻辑在 UI 层配合信令层协同。

### 2. 录制存储方案

- **本地录制**：客户端捕获本地相机、屏幕、声音，采用 H.265（节省存储）或 H.264（兼容性）编码，存储到本地文件系统，支持加密存储和权限管理。
- **服务器录制**：SFU 服务器混流所有参与者的音视频，生成单一输出流，存储到云端（CDN 缓存或对象存储）。支持后续转码、分发、访问权限控制。
- 录制文件格式与转码：支持 MP4/WebM 等格式，可按需进行分辨率、帧率、码率的转码以适应不同客户端或网络环境。

### 3. 安全与合规

- 会议数据端到端加密（E2EE）可选；默认采用传输层加密。
- 支持登录多因子认证、会议密码、动态口令、API 级权限控制。
- 审计日志记录关键操作，便于事后追踪。

### 4. 多端一致性与云协同

- 云端持久化会议状态，支持断点续会和跨端切换（如手机转到桌面）。
- 使用实时数据库（如 Redis 等）同步聊天消息、白板内容。
- 所有端共享统一的业务 SDK，核心逻辑复用率高。

## 五、总结

腾讯会议客户端架构以 **Electron 为核心、Web 技术为表层**，结合深度定制的 WebRTC 内核和原生插件，实现跨平台、高可定制、低延迟的音视频会议体验。关键安全特性包括 Context Isolation 和 Sandbox，屏幕共享采用 desktopCapturer，录制支持本地与服务器双通道。移动端则采用原生实现，但共用同一音视频引擎，保证一致性。前端技术栈强调模块化、TypeScript 和组件化开发，而后端与云服务则提供强大的混流、录制和安全能力，支撑从个人到企业级的多样化会议需求。
